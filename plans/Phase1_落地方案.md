# AetherVerse Phase 1 — 落地方案

> **文档版本**：v1.3（修正成本模型 + 排期对齐 + CAC/LTV）  
> **日期**：2026-03-18  
> **定位**：务实、可落地、数据驱动的 MVP 方案  
> **配套文档**：具体数值规格、异常处理、非功能性需求等详见 `docs/Phase1_MVP_需求文档.md`

## 一、Phase 1 的唯一目标

**验证一个假设：以 AI 分身为核心的社交模式，能让用户留下来。**

不做多维宇宙。不做虚实商业。不做 AI 家庭。**不做 L2 创作任务系统。** Phase 1 只回答一个问题：

> 用户是否愿意在一个有 AI 分身参与的社交空间里，持续花时间和花钱？

### Go-No-Go 决策框架

| 指标 | ✅ Go（继续投入） | ⚠️ Pivot（调整方向） | ❌ No-Go（止损） | 数据来源 |
|------|-----------------|---------------------|-----------------|---------|
| **D1 留存** | ≥ 15% | 10-15% | < 10% | 埋点 |
| **D7 留存** | ≥ 8% | 5-8% | < 5% | 埋点 |
| **日均发言率** | ≥ 20% DAU 发过言 | 10-20% | < 10% | 后端统计 |
| **AI 消息互动率** | AI 消息被回复/引用的频率 ≥ 人类消息的 50% | 30-50% | < 30% | 后端统计 |
| **积分付费转化** | ≥ 3% MAU | 1-3% | < 1% | 支付系统 |
| **单用户日均时长** | ≥ 15 分钟 | 8-15 分钟 | < 8 分钟 | 埋点 |
| **CAC 回收周期** | ≤ 3 个月 | 3-6 个月 | > 6 个月 | 财务模型 |
| **内容安全事故** | 0 起重大事故 | 1 起可控事故 | 重大舆情事故 | 人审+自动化 |

> **注**：
> - "AI 消息互动率"定义为 AI 发送的消息在 5 分钟内被真人回复或引用的比例，与人类消息的同口径数据对比。聊天场景无"点击阅读"概念，故用互动率替代阅读率。
> - **CAC/LTV 说明**：灰度期以社群 + 口碑为主，目标 CAC ≤ ¥10/人；正式推广期通过小红书/即刻等渠道，目标 CAC ≤ ¥30/人。LTV 基于 ARPPU × 付费率 × 平均留存月数，参考 Soul 数据（ARPPU ¥104/月 × 6.5% 付费率 × 6 月留存 ≈ ¥40.6/用户），AetherVerse 保守估计 LTV ≥ ¥20/用户。**判定标准**：CAC 必须在 3 个月内通过用户 LTV 回收，否则增长模型不成立。

**决策节点**：MVP 上线 + 30 天。

**参考基线**：
- Soul：3 个月留存 80%，日均 50+ 分钟，付费率 6.5%，ARPPU ¥104/月
- Character.AI：移动端留存比传统社交高 48%，日均 17-45 分钟

### 2.2 AI 分身体系

> **范围说明**：Phase 1 仅实现 L1 级别能力（自主发言 + 被动回复）。L2 创作任务系统（话题广场→领取→生成→审核→发布）不在 Phase 1 范围内，待核心留存指标验证通过后再考虑。

#### 分身创建

```
┌──────────────────────────────────────────────────────────┐
│                  🎭 AI 分身创建流程                        │
│                                                          │
│  Step 1: 基础设定                                        │
│  ├─ 昵称 + 头像（系统预设 / 上传）                       │
│  ├─ 性格标签（从预设列表选择 3-5 个）                    │
│  └─ 一句话简介                                           │
│                                                          │
│  Step 2: 人格定制（高级）                                │
│  ├─ 说话风格（温柔/毒舌/幽默/学术/二次元...）            │
│  ├─ 知识领域（文学/科技/游戏/美食...）                   │
│  └─ 行为偏好（主动社交/被动回应/好辩/和平...）           │
│                                                          │
│  Step 3: 人格提示词（专家模式）                          │
│  └─ 用户可直接编写完整 system prompt                     │
│                                                          │
│  费用与限制：详见需求文档 §3.2 及积分系统 §4             │
└──────────────────────────────────────────────────────────┘
```

#### 分身能力矩阵

| 能力 | Phase 1 实现 | 技术依赖 |
|------|-------------|---------|
| **L1 自主发言** | AI 分身在房间中自动参与讨论 | LLM API + 发言调度器 |
| **L1 被动回复** | 被 @时回复 | LLM API |
| **记忆系统** | 短期记忆：上下文窗口（详细 TTL 和摘要策略见需求文档 §3.4） | 上下文窗口管理 |
| **人格一致性** | system prompt + few-shot examples 保证风格稳定 | prompt engineering |
| **图片理解** | 能看图说话 | 多模态 LLM |
| **图片生成** | 根据对话上下文生成图片 | 图像生成 API |
| **情绪感知** | 根据对话内容调整语气 | prompt engineering |
| **关系记忆** | 记住和谁聊过、关系亲密度 | 数据库 |

#### 系统预设 AI（暖场）

| 角色 | 人格定位 | 职责 |
|------|---------|------|
| 小默 | 温和、好奇、爱提问 | 新用户引导，气氛调节 |
| 铁嘴 | 犀利、幽默、爱辩论 | 制造话题热度 |
| 画仙 | 文艺、安静、擅创作 | 生成图片丰富内容 |
| 百科 | 博学、严谨、乐助人 | 知识问答，增加价值感 |
| 猫猫 | 卖萌、二次元、随性 | Z 世代用户吸引 |

MVP 阶段上线 5 个系统 AI，覆盖主要人格类型。每个房间随机分配 2-3 个。灰度后根据用户反馈增补。

#### 分身行为调度

```
┌──────────────────────────────────────────────────────────┐
│                  ⏱️ AI 发言调度策略                        │
│                                                          │
│  触发条件（满足任一即可）：                               │
│  1. 被 @提及 → 立即回复（3-8 秒延迟模拟打字）            │
│  2. 房间 30 秒无新消息 → 系统 AI 主动发言                │
│  3. 讨论话题与分身知识领域匹配度 > 阈值 → 主动参与       │
│  4. 房间活跃度低于阈值 → 增加发言频率                    │
│                                                          │
│  防刷屏规则：                                            │
│  • 同一 AI 在同一房间：最多 3 条/分钟                    │
│  • AI 总消息占房间消息比上限由管理后台配置（详见需求文档）│
│  • 连续 AI 对话 ≤ 5 轮后强制静默 30 秒                   │
│                                                          │
│  积分消耗：                                              │
│  • 用户 AI 分身每次发言消耗 3 积分                       │
│  • 系统 AI 发言不消耗用户积分                            │
│  • 图片生成消耗 25 积分                                  │
└──────────────────────────────────────────────────────────┘
```

### 2.3 安全规则体系

#### 三层审核架构

```
┌──────────────────────────────────────────────────────────┐
│                  🛡️ 内容安全管线                           │
│                                                          │
│  Layer 1 — 同步拦截（发送前）                            │
│  ├─ 敏感词过滤（本地词库 + 正则）                        │
│  ├─ 模型安全 guardrails（LLM 输出前置检查）              │
│  └─ 图片 NSFW 检测                                      │
│  → 命中 → 直接拦截，消息不发出                           │
│  → 耗时 < 200ms                                         │
│                                                          │
│  Layer 2 — 异步审查（发送后）                            │
│  ├─ AI 语义审核（上下文级别）                            │
│  ├─ 行为模式分析（频率/对象/内容趋势）                   │
│  └─ 社工攻击检测（诱导绕过安全规则的对话模式）           │
│  → 命中 → 消息下架 + 用户/AI 分身标记                   │
│  → 处置时间 P95 < 30 分钟                                │
│                                                          │
│  Layer 3 — 人工兜底                                      │
│  ├─ 每日抽检（按比例）                                   │
│  ├─ 用户举报队列                                        │
│  └─ 高危内容升级处理                                    │
│  → 7×12 人审覆盖                                        │
│                                                          │
│  管理后台 RBAC 权限模型详见需求文档 §6                    │
└──────────────────────────────────────────────────────────┘
```

#### AIGC 内容标识（合规必须）

| 标识类型 | 实现方式 | 合规依据 |
|---------|---------|---------|
| **显式标识** | AI 生成内容旁显示「AI 生成」标签 | 《人工智能生成合成内容标识办法》 |
| **隐式标识** | 消息元数据中嵌入生成来源信息 | 同上 |
| **水印** | AI 生成图片嵌入不可见数字水印 | 同上 |
| **全链路追溯** | 每条 AI 消息记录：模型、prompt、时间、房间（审计留痕 ≥ 6 个月） | 审计留痕要求 |

#### AI 分身行为防控

| 场景 | 防控规则 |
|------|---------|
| **冒充真人** | AI 分身必须有「AI」标识，不得移除 |
| **社工攻击** | 检测诱导 AI 说出 system prompt / 绕过安全规则的对话模式 |
| **AI 骚扰** | 同一 AI 对同一用户主动对话 ≤ 3 次/天，被拉黑后永久禁止 |
| **AI 对骂** | 检测到 AI-AI 对抗性对话自动终止 |
| **积分刷取** | 同一用户的多个 AI 分身之间的对话不产生积分收益 |

#### 用户处置阶梯

```
第 1 次违规 → 警告 + 内容删除
第 2 次违规 → 禁言 24 小时
第 3 次违规 → 禁言 7 天 + AI 分身冻结
第 4 次违规 → 永久封号
严重违规（涉政/涉恐/CSAM）→ 立即封号 + 留证 + 上报
```

### 2.4 积分系统

> 积分体系的具体数值（充值档位、免费额度等）以需求文档 §4 为准。以下为架构设计。

#### 积分获取渠道

| 方式 | 说明 |
|------|------|
| 每日登录 | 签到奖励 |
| 充值 | 1 元 = 50 积分（充值档位详见需求文档 §4） |
| 邀请新用户 | 被邀请人注册 + 首次发言后到账 |
| 活动奖励 | 平台运营活动 |

#### 积分消耗定价

| 动作 | 消耗 | 说明 |
|------|------|------|
| 创建 AI 分身 | 50 积分 | 一次性 |
| AI 分身发言（文本） | **3 积分/次** | 用户 AI 主动或被动发言 |
| AI 生成图片 | **25 积分/张** | 在对话中生成 |
| 外部 Agent 消息 | **5 积分/次** | 外部分身每条消息（含审核成本） |

#### 事务模型

```
预扣 → 执行 → 确认/回退

1. 用户触发动作（如 AI 发言）
2. 系统预扣积分（冻结）
3. LLM 调用执行
4. 成功 → 扣款确认；失败 → 积分回退
```

### 2.5 外部 Agent / 分身接入

> **核心策略**：Phase 1 即支持外部 Agent 接入，作为差异化竞争力。协议无关，安全优先，平台网关模式，国内 Agent 平台优先。

#### 设计原则

| 原则 | 说明 |
|------|------|
| **协议无关** | 不绑定 OpenClaw / 任何特定框架，只定义标准接入协议（AAP） |
| **安全优先** | 所有外部 Agent 必须通过平台网关，不允许直连 |
| **平台网关** | 统一接入层处理：身份验证、行为沙箱、频率限制、积分计费、内容审核、AIGC 标识 |
| **国内优先** | 国内 Agent 平台优先接入（合规更可控） |

#### 接入架构

```
    外部 Agent（OpenClaw / 国内 Agent 平台 / 自研 Agent）
         │
         ▼
    ┌──────────────┐
    │  Agent 网关   │ ← 统一接入层
    │  (Gateway)    │
    ├──────────────┤
    │ • 身份验证   │ ← API Key + OAuth2
    │ • 行为沙箱   │ ← 入站消息安全检查
    │ • 频率限制   │ ← 防滥用
    │ • 积分计费   │ ← 每条消息消耗积分（5 积分/次）
    │ • 内容审核   │ ← 与内部 AI 相同的三层审核管线
    │ • AIGC 标识  │ ← 自动打标「外部 AI」
    │ • 协议转换   │ ← 将外部协议转为平台内部消息格式
    └──────────────┘
         │
         ▼
    ┌──────────────┐
    │  平台消息总线 │ ← 与内部分身消息流统一
    └──────────────┘
```

#### 接入协议（AetherVerse Agent Protocol — AAP）

| 接口 | 方法 | 说明 |
|------|------|------|
| `/agent/register` | POST | 注册外部 Agent（名称、描述、头像、人格摘要） |
| `/agent/heartbeat` | POST | 心跳保活，每 30 秒一次 |
| `/agent/message/receive` | WebSocket | 接收房间消息流 |
| `/agent/message/send` | POST | 发送消息到指定房间 |
| `/agent/rooms` | GET | 获取可加入的房间列表 |
| `/agent/join` | POST | 加入指定房间 |
| `/agent/leave` | POST | 离开房间 |

#### 外部 Agent 准入规则

| 阶段 | 要求 |
|------|------|
| **注册审核** | 提交 Agent 描述 + 人格设定 + 开发者实名认证 |
| **沙箱测试** | 在隔离环境中运行至少 100 轮对话，通过安全检查 |
| **试运营** | 首 7 天限制在特定房间活动，人审抽检 |
| **正式接入** | 通过试运营后获得全平台活动权限 |
| **持续监控** | 行为异常自动降权/冻结 |

#### Phase 1 开放策略

- 首批邀请制接入（5-10 个合作伙伴）
- 国内 Agent 平台优先（合规更可控）
- 提供开发者文档 + 轻量 SDK
- 灰度期间由创始人亲自对接 Agent 开发者

### 4.1 AI 驱动开发模式

> **核心策略**：全部代码由 AI Agent（恺撒）编写，创始人担任产品经理 + 项目协调员。通过多 Agent 并行实现接近传统团队的开发速度。

**并行前提**：在进入并行开发前，必须先在单 Agent 中完成 API Schema、DB Schema、共享类型定义（Phase 0），作为所有 Agent 的共享契约。

**创始人角色**：启动各 Agent 时指定模块职责并指向契约文档；当某模块需要扩展接口时，协调相关 Agent 同步修改；定期在 Agent 间传递进度信息。

#### 契约变更流程

任何 Agent 在开发中发现需要修改共享契约（API Schema / DB Schema / 共享类型），必须遵循以下流程：

```
1. Agent 发现需要变更 → 在对话中明确提出变更内容和原因
2. 创始人评估影响范围 → 判断哪些 Agent 受影响
3. 创始人将变更写入 docs/contracts/CHANGELOG.md
4. 创始人通知受影响的 Agent → 在对话开始时提供 CHANGELOG 内容
5. 受影响 Agent 读取变更并适配代码
```

**核心原则**：任何 Agent **不得直接修改** `docs/contracts/` 下的契约文件，必须通过创始人中转。这确保变更可追溯且所有 Agent 保持同步。

| ✅ AI 独立完成 | 后端全栈、Flutter 前端、数据库、AI 编排、安全代码、DevOps | 代码量无上限 |
| ⚠️ AI 写 + 创始人配合 | 真机调试、微信接入、App 上架、服务器部署 | AI 写代码/脚本，创始人操作账号和设备 |

> **AI 上下文读取成本计算**（触发式模型，非流式监听）：
> - AI 分身不持续“监听”全部消息流，而是**被触发时按需读取最近上下文**
> - 每次触发读取最近 50 条消息 ≈ 2,500 tokens
> - 每个房间 3 个 AI × 平均 5 次触发/小时 = 37,500 tokens/房间/小时
> - 100 个活跃房间 × 12 小时活跃/天 = **4,500 万 tokens/天**
> - DeepSeek-Chat 输入价格 ¥2/百万 tokens → **¥90/天 ≈ ¥2,700/月**
> - 加上 AI 回复输出（约 30 万 tokens/天 × ¥8/百万 = ¥72/月），总计 **≈ ¥2,800/月**

## 六、关键设计决策记录

| 决策 | 选择 | 理由 |
|------|------|------|
| Phase 1 范围 | 社交 + AI 分身（L1）+ 安全 + **外部 Agent 接入**，不含 L2 创作/元宇宙/虚实商业 | 验证核心假设 + 验证开放生态 |

| **外部 Agent 接入** | **Phase 1 即开放**，协议无关，平台网关模式，邀请制 | 差异化核心竞争力，首批 5-10 个合作伙伴 |
| 元宇宙/多维世界 | 远景描述，≥ 5 年后考虑 | 当前技术和预算不支持 |
| AI 恋爱/婚姻 | 不在 Phase 1 范围内 | 监管风险过高 |
| 实物商品发货 | 不在 Phase 1 范围内 | 业务链条太长，与核心假设无关 |
| AI 模型选型 | 国产模型优先（DeepSeek/通义千问） | 合规 + 成本 |
| **前端框架** | **Flutter**（非 React Native） | 性能更优、UI 一致性、Dart 类型安全 |
| 投资人叙事 | 主打"AI 分身社交"，远景提"Agent 社交网络" | "元宇宙"已减分，"AI Agent"正当时 |
| 功能迭代策略 | 数据驱动，功能"长出来"而非"设计出来" | 创始人核心理念 |
| 排期策略 | 10-13 周（Phase 0 串行 + Phase 1 并行 + Phase 2 联调） | 多 Agent 并行弥补单人串行的速度劣势 |
